Building an end-to-end ML solution in less than an hour

Motivation

In a previous article, Predicting Medical Charges using AzureML and SkLearn Linear Regression, I created a regression model. There was certainly a large number of lines of code. AzureML offers a drag-and-drop interface called AzureML Designer. The experience and feel are similar to RapidMiner.

Scenario

Using the common Titanic Dataset, create a classification model to predict if a passenger survived the 1912 Titanic Accident. Perhaps the analysis can be used for a safety review.

I will use Age, Sex, and Ticket Fare to predict if the passenger survived or not.

Azure Designer Pipeline

Create Data Asset

After connecting to the AzureML Workspace, let’s create a data asset from the raw CSV downloaded from Kaggle.

default_ds = ws.get_default_datastore()

if 'titanic dataset' not in ws.datasets:
    Dataset.File.upload_directory(src_dir='data',
                              target=DataPath(default_ds, 'titanic-data/')
                              )

    #Create a tabular dataset from the path on the datastore (this may take a short while)
    tab_data_set = Dataset.Tabular.from_delimited_files(path=(default_ds, 'titanic-data/titanic.csv'))

    # Register the tabular dataset
    try:
        tab_data_set = tab_data_set.register(workspace=ws, 
                                name='titanic dataset',
                                description='titanic data',
                                tags = {'format':'CSV'},
                                create_new_version=True)
        print('Dataset registered.')
    except Exception as ex:
        print(ex)
else:
    print('Dataset already registered.')

The Data Asset can be found under the ‘Data’ blade of AzureML

Configure Compute for Pipeline

Compute is needed for the pipeline to process the necessary steps for data processing and training.

from azureml.core.compute import ComputeTarget, AmlCompute
from azureml.core.compute_target import ComputeTargetException

cluster_name = "sweetdreams"

try:
    # Check for existing compute target
    pipeline_cluster = ComputeTarget(workspace=ws, name=cluster_name)
    print('Found existing cluster, use it.')
except ComputeTargetException:
    # If it doesn't already exist, create it
    try:
        compute_config = AmlCompute.provisioning_configuration(vm_size='STANDARD_DS11_V2', max_nodes=2)
        pipeline_cluster = ComputeTarget.create(ws, cluster_name, compute_config)
        pipeline_cluster.wait_for_completion(show_output=True)
    except Exception as ex:
        print(ex)

Now we’re ready to build our pipeline.

Access the Designer

Select ‘Designer’ Blade from the AzureML UI and select ‘ Create a new pipeline using classic prebuild components’

With the compute cluster provisioned, we must attach it to the pipeline via settings.

Create and Run the Training Pipeline

Below is a very simple pipeline for an end-to-end classification model.

Detailed Description of Steps:

Note: Component Parameters can be found in my Github images

  1. Titanic dataset
    - Retrieves the datstore that contains our raw data. Datastore
  2. Select Columns in Dataset
    - From the base dataset, select columns of interest such as features (our predictors) and label (what we’re trying to predict) column
  3. Clean Missing Data
    - Perform Imputation for missing data or opt to remove the row completely from the pipeline
  4. Normalize Data
    - Perform Normalization such as MinMax Standardization on continuous variables (Ex. Age and Fare)
    - This step is not necessary for Decision Tree algorithms, but included for example
    - Typically used for distance-based algorithms or deep learning models, for accuracy and performance
  5. Split Data
    - Split the data into training and test set.
    - You can define random seed and proportion of test set Ex. (.2 or 20% of the data is held out for testing
    - From here, the nodes start to diverge where the training set would be connected to the ‘Train Model’ and test data sent to ‘Score Model’
  6. Two-Class Boosted Decision Tree
    - The selected algorithm to perform machine learning.
    - Hyperparameters can be entered here.
    - No hyperparameter tuning in this example.
  7. Train Model
    - From the Selected Algorithm and Training Data, perform Model Training where it reaches its objective threshold or the number of iterations has been completed
    - The Model with its trained weights will be stored as an artifact in AzureML.
  8. Score Model
    - From the Trained Model, run predictions using training and test set.
  9. Evaluate Model
    - compare the results based on a selected evaluation metric such as AUC, Precision, Recall, or Accuracy.

Create and Run the Inference Pipeline

After we’re satisfied with the training pipeline, select ‘Create Inference Pipeline’ and AzureML will automatically set up the components.

For the pipeline to work, you would need to remove components such as ‘Evaluate Model’, ‘Select Columns in Dataset’, and the first transformation components because they are not relevant for inferences and will create an error. You would also need a few components to allow inferencing.

The Execute Python Script has the following code to handle the returning prediction.

Create the Endpoint

Once we’re satisfied with the Inference Pipeline, we select ‘Deploy’ and AzureML will start to provision the endpoint.

Make Inferences

After the endpoint is successfully created, you can run inferences! The Input Date from ‘Test’ was provided by us by the ‘Enter Data Manually’ component of the inference pipeline.

You can use the ‘Consume’ Tab to get the servicing URL and API Keys. Since this is a test endpoint, it is accessible to the public internet, but for production load, you would want to consider an application firewall and enough compute to handle anticipated requests.

Summary/Takeaways/Conclusion

  1. Faster Proof of Concept
    - Although AzureML offers AutoML to take raw data and automatically run data preparation and algorithms, you might already know your approach or the business has already decided a model and algorithm.
    - You could use AzureML Designer to quickly spin up a proof of concept before committing development hours to a notebook.
  2. Utilize the logs
    - Pipeline steps offer outputs and logs where can inspect logs and data previews. I found this incredibly useful to see if the pipeline is running as expected or if there are error logs, allowing an opportunity to find a root cause. This granular view helps in troubleshooting as opposed to scanning an entire notebook or python script!
  3. Needing Microsoft to adopt exportability for the AzureML Designer
    - Currently, pipelines can be saved within workspaces and URLs can be shared. But there is no way to export the pipeline configuration or even a PDF. If Microsoft can package pipelines into a lightweight form such as JSON or XML, then pipelines can be more operatable between organizations and Azure Workspaces.
  4. Inferring the Market
    -     Based on conversations with Data Scientists in the field, Machine Learning Workloads are increasingly operationalized and applications in the market are more accessible to non-technical. There is more offering for more GUI-based Machine Learning or the application writes the code for you based on given parameters. I do not expect Data Scientists to be out of the job because they are required to run the workload and interpret the model and recommend decisions to stakeholders.
